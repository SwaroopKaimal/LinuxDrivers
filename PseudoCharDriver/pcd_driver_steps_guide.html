THE ONE MENTAL MODEL — USER SPACE vs KERNEL SPACE
›
User space calls open/read/write/lseek/close — these are C library wrappers around CPU instructions that trigger a software interrupt (syscall). The CPU switches privilege level and jumps into kernel mode.
›
VFS (Virtual File System) is the kernel's switchboard. It looks up the file_operations struct registered for this device and calls the matching function pointer.
›
Your driver is just a set of C functions registered into that struct. The kernel calls them — you never call them yourself.
›
Crossing the boundary: user pointers cannot be dereferenced in kernel space — use copy_to_user / copy_from_user. This is a favourite interview question.
1
Define your data structures first
before any code
THINK: what does each device need to know about itself?
1.1
Per-device struct — pcdev_private_data
Holds everything one device instance needs: char *buffer, unsigned size, serial_number, perm, and crucially an embedded struct cdev. The embedded cdev is how you later recover this struct using container_of in pcd_open.
interview: why embed cdev inside private_data?
1.2
Driver-wide struct — pcdrv_private_data
Holds the base dev_t device_number, a pointer to struct class, pointer to struct device, and an array of all per-device structs. One global instance of this — the driver's single source of truth.
1.3
Define permissions as readable constants
Use #define RDONLY 0x01, WRONLY 0x10, RDWR 0x11. These bitmasks map neatly: reading checks bit 0, writing checks bit 4. Interviewers will ask you to explain why these values were chosen.
concept: bitmask permission design
2
Module init — register everything with the kernel
the 4-step init sequence
REMEMBER THIS ORDER — cleanup reverses it exactly
2.1
Allocate device numbers — alloc_chrdev_region()
Ask the kernel for N consecutive minor numbers. It picks an available major and gives you back a dev_t base. Never hardcode a major number — that's a legacy practice.
API: alloc_chrdev_region(&dev_num, 0, N, "name") pitfall: check return value — negative = failure
2.2
Create a device class — class_create()
Creates an entry under /sys/class/. This is needed so device_create can later auto-create /dev nodes via udev. Returns a pointer — check with IS_ERR(), not NULL check.
API: class_create("pcd_class") pitfall: use IS_ERR() not == NULL for pointer errors
2.3
For each device — cdev_init + cdev_add
cdev_init binds your file_operations struct to the cdev. cdev_add makes it live — from this moment, the kernel can dispatch syscalls to your driver. Do this inside a loop over all N devices.
API: cdev_init(&cdev, &fops) → cdev_add(&cdev, dev_num+i, 1) interview: what happens between cdev_init and cdev_add?
2.4
For each device — device_create()
Populates /sys/class/pcd_class/pcdev-N and triggers udev to create /dev/pcdev-N automatically. Without this, no /dev node appears and the user cannot open the device.
API: device_create(class, NULL, dev_num+i, NULL, "pcdev-%d", i+1)
2.5
Error handling with goto labels
On any failure, jump to a cleanup label that undoes only the steps completed so far. This is the standard kernel pattern — goto unregister_chrdev, goto cdev_del, etc. Interviewers love this: it's resource-safe and avoids nested ifs.
interview: why goto in kernel code is acceptable and preferred
3
Implement file operations — the 5 driver functions
what VFS calls on each syscall
REMEMBER: parameters come from VFS — you never call these directly
3.1
pcd_open — the gatekeeper
Three jobs in order:
a) Get minor number from inode->i_rdev using MINOR()
b) Recover per-device struct using container_of(inode->i_cdev, struct pcdev_private_data, cdev)
c) Save it to filp->private_data so read/write/lseek can find it
d) Call check_permission() — reject if mode mismatch
concept: container_of is the key to per-device data pitfall: your code returns 0 even on permission failure — must return ret
3.2
pcd_read — copy kernel buffer → user
a) Get max_size from filp->private_data
b) Clamp count: if f_pos + count > max_size, reduce count
c) copy_to_user(buff, device_buf + f_pos, count)
d) Advance *f_pos += count
e) Return bytes read (0 = EOF signal to user)
interview: why can't you use memcpy instead of copy_to_user?
3.3
pcd_write — copy user → kernel buffer
a) Clamp count to remaining space
b) If count == 0 after clamping, return -ENOMEM (buffer full)
c) copy_from_user(device_buf + f_pos, buff, count)
d) Advance *f_pos += count
e) Return bytes written
pitfall: -ENOMEM is wrong here — -ENOSPC is more accurate, but -ENOMEM is common in examples
3.4
pcd_lseek — reposition the cursor
Handle 3 cases: SEEK_SET (absolute), SEEK_CUR (relative to current), SEEK_END (relative to end). For each: compute new position, validate it is within [0, max_size], update filp->f_pos, return new position.
interview: what does lseek return on success? (new absolute offset)
3.5
pcd_release — called on close()
Called when the last fd referencing this open file is closed. In this driver — nothing to do: no dynamic allocations were made in open, no hardware to release. Just log and return 0. In real drivers this is where you free resources.
4
Module exit — undo everything in reverse order
mirror image of init
RULE: last thing created must be first thing destroyed
4.1
Loop: device_destroy + cdev_del for each device
Removes /dev/pcdev-N and the sysfs entry, then unregisters the cdev from VFS. After cdev_del, no new syscalls will reach your driver.
4.2
class_destroy → unregister_chrdev_region
Remove the class from sysfs, then release the device numbers back to the kernel pool. This order matters: destroying the class first ensures no device nodes try to reference it during teardown.
interview: what happens if you skip unregister_chrdev_region?
5
Interview flashpoints — questions you will be asked
memorise these answers
Q1
What is the difference between a major and minor number?
Major identifies the driver. Minor identifies a specific device managed by that driver. Your 4 pcdevs share one major but have minors 0–3.
Q2
Why can't you use memcpy to transfer data between user and kernel?
User-space pointers may be invalid, swapped out, or malicious. copy_to_user / copy_from_user perform access validation and handle page faults safely. Direct dereference of a __user pointer can panic the kernel.
Q3
What is container_of and why is it used here?
A macro that computes the address of a parent struct given a pointer to one of its members. In pcd_open, VFS gives you inode->i_cdev. Since cdev is embedded inside pcdev_private_data, container_of lets you recover the whole private struct from just that pointer.
Q4
What is the role of filp->private_data?
A void pointer in struct file that is yours to use. Set it in pcd_open to the per-device struct. All subsequent calls (read/write/lseek) receive the same filp, so they cast private_data to get device-specific buffer, size, and permissions — without needing the inode again.
Q5
What is the difference between cdev_init and cdev_add?
cdev_init sets up the cdev struct in memory (links it to file_operations). cdev_add registers it with the kernel's character device database — making it live. A device is not accessible until cdev_add returns successfully.
Q6
What is the role of VFS in a character driver?
VFS is the kernel abstraction that makes every file, socket, pipe, and device look the same to user space. When a process calls read(), VFS looks up the file_operations for that fd and dispatches to the driver's .read function pointer. Your driver never cares what syscall ABI the architecture uses.
Q7
Why is goto used in kernel init functions?
Kernel code cannot use C++ exceptions or RAII. goto to cleanup labels is the accepted pattern for unwinding partial initialisation — it's readable, flat, and guarantees each resource is freed exactly once in the right order. It is explicitly recommended in the Linux kernel coding style guide.
